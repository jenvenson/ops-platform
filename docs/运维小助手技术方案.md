# 运维小助手技术方案

## 1. 文档目标

本文档用于设计 OPS Platform 内部“大模型运维小助手”能力，目标是在当前平台基础上提供可落地的智能问答、页面导航、运维查询和受控执行能力。

设计原则：

- 先做平台内助手，不做通用聊天机器人
- 先做只读问答与查询，再做执行操作
- 模型只负责理解和生成，不直接连接生产系统
- 所有业务访问统一走受控工具层
- 高风险操作必须鉴权、确认、审计

## 2. 当前现状

当前项目中已有一个基础聊天入口：

- 前端：[frontend/src/components/AIChatbot.tsx](frontend/src/components/AIChatbot.tsx)
- 后端：[backend/internal/assistant/](backend/internal/assistant)

当前机器人实际能力：

- 前端提供聊天浮窗
- 后端提供 `/api/assistant/sessions`、`/api/assistant/messages` 以及历史查询接口
- 回复逻辑为 `RAG + 只读工具 + Ollama(qwen2.5:1.5b)`，模型不可用时回退到受控 fallback

当前问题：

- 不能理解复杂问题
- 不能结合文档和实时数据回答
- 不能调用平台已有业务能力
- 不能区分“查询类”和“执行类”请求
- 缺少权限、确认、审计闭环

当前 `assistant` 模块已经成为主链路，后续设计以继续增强这条链路为准。

## 3. 建设目标

### 3.1 一期目标

一期聚焦“智能问答 + 页面导航 + 平台只读查询”：

- 回答平台使用类问题
- 根据问题返回页面导航路径
- 查询 CMDB、部署、告警、监控、安全等模块的只读信息
- 基于知识库回答“怎么做”“在哪里做”“为什么失败”
- 返回结构化引用来源

### 3.2 二期目标

- 增加受控执行能力
- 支持发布、重试、回滚、配置同步等操作
- 增加二次确认、审批和执行审计

### 3.3 三期目标

- 增加主动建议和诊断能力
- 支持连续多轮上下文分析
- 支持事件聚合、根因排查和 SOP 推荐

## 4. 总体架构

```text
前端 AIChatbot
  -> Assistant API
    -> 会话编排层
      -> 意图识别
      -> 权限校验
      -> 知识检索
      -> 工具选择
      -> 大模型生成
    -> 平台工具层
      -> CMDB 工具
      -> 部署工具
      -> Jenkins 工具
      -> 告警工具
      -> 监控工具
      -> 安全工具
      -> 页面导航工具
    -> 审计与日志
```

核心思想：

- 模型不直接访问数据库、Jenkins、Consul、Grafana
- 模型只输出“理解结果”和“工具调用意图”
- 真正的数据访问由后端工具层执行

## 5. 模块划分

建议在后端新增：

```text
backend/internal/assistant/
├── api/                      # HTTP Handler
├── service/                  # 会话编排与主流程
├── llm/                      # 模型客户端封装
├── retriever/                # 知识检索
├── tools/                    # 受控工具注册与调用
├── prompt/                   # 系统提示词模板
├── memory/                   # 会话上下文管理
├── audit/                    # 审计日志
└── models/                   # assistant 相关数据结构
```

各模块职责：

- `api/`
  对外暴露 `/api/assistant/*` 接口
- `service/`
  负责主链路编排：识别意图、选工具、生成回复
- `llm/`
  屏蔽具体模型供应方，可支持 OpenAI、Ollama、内部模型
- `retriever/`
  负责知识分片、向量检索、引用合并
- `tools/`
  将平台能力封装成标准工具接口
- `memory/`
  保存有限轮次上下文，不让 prompt 无限膨胀
- `audit/`
  记录用户问题、工具调用、执行结果、风险等级

## 6. 核心能力设计

### 6.1 问答类能力

适用问题：

- 如何创建部署任务
- 告警规则在哪配置
- Jenkins 视图管理在哪里
- 聚合打包失败怎么看日志

处理方式：

1. 识别为“知识问答”
2. 检索 `docs/`、用户手册、页面路径、模块说明
3. 拼接引用上下文
4. 调用模型生成简洁回答
5. 返回引用来源和页面路径

### 6.2 查询类能力

适用问题：

- 昨晚哪个发布失败了
- 某项目测试环境有哪些机器
- 最近有哪些高危漏洞
- 当前有哪些未恢复告警

处理方式：

1. 识别为“业务查询”
2. 将问题转成受控工具参数
3. 调用业务工具获取实时数据
4. 模型对结果做摘要和解释
5. 返回结构化结果与建议入口

### 6.3 执行类能力

适用问题：

- 触发某应用重新发布
- 同步某环境配置
- 重试某 Jenkins 构建

执行规则：

- 必须鉴权
- 必须判断风险等级
- 高风险动作必须二次确认
- 必须记录执行审计

一期不落地执行类，只预留架构。

## 7. 工具层设计

所有平台能力以标准工具形式暴露给 Assistant，不允许模型自由拼接 SQL、Shell 或 HTTP。

### 7.1 工具接口建议

```go
type Tool interface {
    Name() string
    Description() string
    InputSchema() any
    Authorize(ctx *ToolContext, input []byte) error
    Execute(ctx *ToolContext, input []byte) (any, error)
}
```

### 7.2 一期工具清单

#### 页面导航工具

- `navigate_to_page`
- 输入：功能名称、模块名称
- 输出：页面标题、路由路径

#### CMDB 查询工具

- `query_projects`
- `query_environments`
- `query_servers`
- `query_applications`

#### 部署查询工具

- `query_release_history`
- `query_aggregate_history`
- `query_jenkins_build_status`

#### 告警查询工具

- `query_alert_events`
- `query_alert_rules`
- `query_alert_contacts`

#### 监控查询工具

- `query_monitor_overview`
- `query_dashboard_links`

#### 安全查询工具

- `query_security_tasks`
- `query_vulnerabilities`
- `query_assets`

### 7.3 二期工具清单

- `trigger_release`
- `retry_release`
- `rollback_release`
- `sync_consul_config`
- `retry_jenkins_job`

这些工具必须支持操作确认和权限分级。

## 8. 知识库设计

一期知识源建议只接入高价值、稳定资料：

- [docs/deploy.md](docs/deploy.md)
- [docs/design.md](docs/design.md)
- [docs/testing.md](docs/testing.md)
- [docs/user_manual.md](docs/user_manual.md)
- [docs/project-structure.md](docs/project-structure.md)
- 菜单配置与页面路由
- 常见失败记录与修复文档

### 8.1 处理方式

- 对 Markdown / HTML 文档做分片
- 每个分片保留标题、路径、模块标签
- 建立向量索引
- 检索后附带来源路径和片段摘要

### 8.2 返回要求

模型回答时建议包含：

- 简短结论
- 操作步骤
- 页面入口
- 来源引用

## 9. 模型接入设计

模型接入应抽象成统一客户端，不在业务层写死某一供应方。

建议接口：

```go
type ChatModel interface {
    Chat(ctx context.Context, req ChatRequest) (*ChatResponse, error)
}
```

支持的后端可以包括：

- OpenAI 兼容接口
- Ollama
- 企业内部模型网关

### 9.1 模型职责边界

模型负责：

- 理解问题
- 判断意图
- 决定是否需要工具
- 整理最终回答

模型不负责：

- 直接访问数据库
- 直接访问生产 API
- 直接执行命令

## 10. 模型选型与成本建议

结合当前项目阶段，建议采用“默认轻量模型 + 复杂请求升级 + 独立向量模型”的组合。

### 10.1 推荐组合

#### 默认对话模型

- `gpt-5-mini`

适用场景：

- 平台使用问答
- 页面导航
- 常规 RAG 问答
- 简单只读查询结果整理
- 常规工具调用编排

选择原因：

- 响应速度更快
- 成本更可控
- 足够支撑一期“知识问答 + 页面导航 + 只读查询”

#### 升级模型

- `gpt-5.2`

适用场景：

- 多工具联合查询
- 长上下文文档归纳
- 复杂问题拆解
- 需要更稳定推理质量的诊断类问题

选择原因：

- 通用能力更强
- 适合二期以后更复杂的 Agent 编排
- 可作为复杂请求兜底模型

#### 向量模型

- `text-embedding-3-small`

适用场景：

- `docs/` 文档检索
- 用户手册检索
- 故障处理 SOP 检索
- 页面说明与功能说明检索

选择原因：

- 成本低
- 一期知识库规模不大，足够使用
- 后续如中文检索效果不足，再升级到 `text-embedding-3-large`

### 10.2 不建议的一期选择

- `gpt-5.2-pro`
  成本过高，不适合作为日常助手主链路默认模型

- 本地 `Ollama` 作为默认主模型
  当前项目更需要稳定上线和快速迭代，而不是本地模型运维复杂度

- 单一大模型覆盖全部请求
  会让简单问题成本过高，也不利于后续精细化路由

### 10.3 成本控制建议

#### 1. 按问题类型路由模型

- 简单问答和导航：走 `gpt-5-mini`
- 复杂分析问题：升级到 `gpt-5.2`
- 纯结构化查询：优先工具直出，尽量减少模型生成长度

#### 2. 限制输出长度

- 一期默认回答控制在 150 到 300 中文字
- 查询类问题先给摘要，再提供“展开详情”

#### 3. 缩短上下文

- 只带最近 10 到 20 轮消息
- 历史消息摘要化
- 检索只注入命中的文档片段，不全量拼接

#### 4. 让工具先算，模型后说

- 数据筛选、排序、统计由后端工具层完成
- 模型只负责解释和整理
- 避免让模型处理大批原始 JSON

### 10.4 本地模型策略

如果未来有数据出域限制或成本约束，可在二期以后增加本地模型模式：

- 默认模式：云模型
- 降级模式：本地模型
- 特定环境：内网专用模型网关

但本地模型接入建议作为可替换实现，不要反向驱动一期架构。

### 10.5 一期最终建议

一期推荐模型方案如下：

- 对话模型默认：`gpt-5-mini`
- 复杂请求升级：`gpt-5.2`
- 知识库向量：`text-embedding-3-small`

这套组合在效果、速度、成本和工程复杂度之间最均衡，最适合当前 OPS Platform 的“运维小助手”起步阶段。

## 11. 会话编排设计

### 11.1 主流程

```text
用户问题
-> 加载用户信息与角色
-> 读取最近 N 轮会话上下文
-> 意图识别
-> 判断路由:
   - 普通知识问答
   - 实时数据查询
   - 执行动作
-> 如需检索，调用 retriever
-> 如需实时数据，调用 tools
-> 汇总上下文
-> LLM 生成最终回答
-> 落库消息、引用、工具调用日志
```

### 11.2 意图分类

建议至少区分：

- `knowledge_qa`
- `page_navigation`
- `readonly_query`
- `action_request`
- `fallback`

### 11.3 上下文策略

- 只保留最近 10 到 20 轮
- 历史消息做摘要，不直接全量拼入 prompt
- 重要上下文可单独存 `conversation_summary`

## 12. 接口设计

建议逐步将旧 `chatbot` 接口升级到新的 `assistant` 接口。

### 12.1 创建会话

```http
POST /api/assistant/sessions
```

请求：

```json
{
  "scene": "web",
  "user_agent": "Mozilla/5.0"
}
```

响应：

```json
{
  "session_id": "asst_xxx",
  "created_at": "2026-03-12T12:00:00Z"
}
```

### 12.2 发送消息

```http
POST /api/assistant/messages
```

请求：

```json
{
  "session_id": "asst_xxx",
  "message": "如何查看某次部署失败原因？"
}
```

响应：

```json
{
  "message_id": "msg_xxx",
  "answer": "可以在发布历史中查看失败记录，并进入对应任务日志页面。",
  "intent": "knowledge_qa",
  "citations": [
    {
      "title": "部署说明",
      "path": "docs/deploy.md"
    }
  ],
  "actions": [
    {
      "type": "navigate",
      "label": "打开发布历史",
      "path": "/deploy/history"
    }
  ]
}
```

### 12.3 二次确认

```http
POST /api/assistant/actions/confirm
```

用于二期执行类工具确认，不在一期启用。

## 14. 安全设计

### 14.1 权限控制

- Assistant 必须绑定当前登录用户
- 工具调用必须基于当前用户角色校验
- 不允许匿名执行操作类请求

### 14.2 风险分级

- `low`：文档问答、页面导航
- `medium`：只读实时查询
- `high`：发布、回滚、配置变更、任务重试

### 14.3 审计要求

- 所有工具调用留痕
- 所有执行类动作记录确认人
- 对敏感内容做脱敏，如 Token、密码、密钥

## 15. 前端设计建议

前端保留现有聊天浮窗，但建议扩展返回渲染能力。

推荐支持：

- 纯文本回答
- 来源引用列表
- 页面跳转按钮
- 结构化结果卡片
- 执行动作确认框
- 历史会话查看

建议对回答结果使用统一结构：

```ts
type AssistantMessage = {
  id: string
  role: 'user' | 'assistant'
  text: string
  citations?: Array<{ title: string; path: string }>
  actions?: Array<{ type: string; label: string; path?: string }>
  toolCalls?: Array<{ tool: string; status: string }>
}
```

## 16. 落地步骤

### 16.1 Phase 1

- 新建 `backend/internal/assistant/`
- 新建 `POST /api/assistant/sessions`
- 新建 `POST /api/assistant/messages`
- 接入文档检索
- 接入页面导航工具
- 接入只读查询工具
- 前端聊天组件切到新接口

### 16.2 Phase 2

- 接入模型工具调用能力
- 新增审计日志
- 支持结构化动作返回
- 增加执行类工具确认机制

### 16.3 Phase 3

- 增加上下文摘要
- 增加事件诊断和 SOP 推荐
- 增加多数据源联合分析

## 17. 一期验收标准

- 能回答平台常见使用问题
- 能返回正确页面导航
- 能查询主要业务模块只读数据
- 回答带来源引用
- 工具调用有日志
- 没有绕过权限直接访问业务系统

## 18. 结论

运维小助手不应设计成一个“能闲聊的机器人”，而应设计成一个“基于模型理解能力、基于工具访问平台能力、基于审计保障安全边界”的平台内智能入口。

对当前项目而言，最合理路径是：

1. 保留现有聊天 UI
2. 新建独立 `assistant` 模块
3. 先做知识问答和只读查询
4. 再做受控执行

这样可以在不破坏现有业务结构的前提下，逐步把“运维小助手”做成一个真正能落地的系统能力。
