# 线上菜单初始化记录

本文档记录 2026-03-12 一次线上菜单与开发环境不一致的排查和修复过程，供后续重复问题处理时直接参考。

## 1. 问题现象

线上环境发布最新代码后，菜单管理与开发环境不一致，具体表现为：

- 缺少 `安装包聚合`
- 缺少 `聚合历史`
- 缺少 `资产中心`
- 缺少 `漏洞工单`
- `安全概览` 路径与开发环境不一致
- 新补的两个部署菜单标题出现乱码

## 2. 根因结论

问题根因不是前端代码，而是线上数据库中的菜单数据没有跟随代码一起同步。

当前菜单来源链路：

```text
前端 MainLayout
-> GET /api/user/menus
-> 后端 auth.GetCurrentUserMenus
-> 数据库 tables: menus / role_menus
```

因此：

- 发布代码不会自动同步 `menus`
- 发布代码不会自动同步 `role_menus`
- 浏览器本地还会缓存 `user_menus`

## 3. 线上与开发环境差异

本次排查确认线上缺失以下菜单项：

- `deploy-aggregate-package`
- `aggregated-history`
- `security-assets`
- `security-tickets`

同时存在以下配置不一致：

- `security-overview`
  开发环境路径：`/security/overview`
  线上路径：`/security`

线上还存在一个历史遗留禁用菜单：

- `cmdb-idc`

这个项不是本次问题主因。

## 4. 已落库的脚本

本次用于补菜单和补角色授权的脚本已保存在：

- [deploy/sql/sync_online_menus.sql](/Users/edy/Data/code/claude/ops-platform/.worktrees/phase1-framework/deploy/sql/sync_online_menus.sql)

作用包括：

- 补齐缺失菜单
- 修正 `security-overview` 路径
- 给 `admin` / `ops` 补齐权限

## 5. 线上执行方式

标准执行命令：

```bash
docker exec -i ops-mysql mysql -uroot -proot123 ops_platform < /opt/ops-platform/deploy/sql/sync_online_menus.sql
```

注意：

- 本次线上 `role_menus` 表结构只有 `created_at`，没有 `updated_at`
- 如果线上表结构仍然保持当前状态，原脚本中的 `role_menus` 插入部分需要按线上结构适配

## 6. 本次实际执行过程

### 6.1 先补菜单

本次首次执行脚本时，前半段菜单插入和路径修正已经成功执行，后半段在 `role_menus.updated_at` 处失败。

- `admin`

## 7. 本次修正的菜单标题乱码

本次还发现线上新增的两个菜单标题出现乱码，后续已直接在线上修正：

- `deploy-aggregate-package` -> `安装包聚合`
- `aggregated-history` -> `聚合历史`

修正示例：

```sql
UPDATE menus SET title='安装包聚合' WHERE `key`='deploy-aggregate-package';
UPDATE menus SET title='聚合历史' WHERE `key`='aggregated-history';
```

建议执行 SQL 时显式指定：

```bash
mysql --default-character-set=utf8mb4
```

## 8. 线上验证方式

### 8.1 查菜单

```sql
SELECT menus.key, title, path, parent_id, sort, status
FROM menus
WHERE menus.key IN (
  'deploy-aggregate-package',
  'aggregated-history',
  'security-assets',
  'security-tickets',
  'security-overview'
)
ORDER BY parent_id, sort, id;
```

### 8.2 查角色权限

```sql
SELECT r.code, m.`key`, m.title
FROM role_menus rm
JOIN roles r ON r.id = rm.role_id
JOIN menus m ON m.id = rm.menu_id
WHERE m.`key` IN (
  'deploy-aggregate-package',
  'aggregated-history',
  'security-assets',
  'security-tickets'
)
ORDER BY r.code, m.sort, m.id;
```

## 9. 前端缓存处理

即使数据库已经修正，用户端如果仍显示旧菜单，通常是 `localStorage` 缓存未刷新。

处理方式：

1. 退出重新登录
2. 或浏览器控制台执行：

```js
localStorage.removeItem('user_menus')
location.reload()
```

## 10. 下次遇到同类问题的处理顺序

建议固定按这个顺序处理：

1. 对比开发环境和线上 `menus`
2. 对比开发环境和线上 `role_menus`
3. 执行 [deploy/sql/sync_online_menus.sql](/Users/edy/Data/code/claude/ops-platform/.worktrees/phase1-framework/deploy/sql/sync_online_menus.sql)
4. 如果 `role_menus` 字段不一致，先 `DESCRIBE role_menus`
5. 按线上真实字段补执行权限 SQL
6. 修正乱码标题
7. 清理浏览器 `user_menus` 缓存

## 11. 经验结论

后续凡是新增菜单类功能上线，都不要只做代码发布，还要同时确认：

- `menus` 是否已同步
- `role_menus` 是否已同步
- 线上库表结构是否与开发预期一致
- 浏览器缓存是否已刷新

否则容易出现“代码已上线，但菜单没出现”的假象。
