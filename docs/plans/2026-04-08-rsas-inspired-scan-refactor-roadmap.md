# 参考绿盟思路的安全扫描重构路线图

日期：2026-04-08

## 目标

在保留当前可用扫描能力的前提下，按成熟漏洞扫描平台的思路，重构主机漏洞扫描与网站漏洞扫描，使安全中心从“扫描任务页面”升级为“资产发现 + 风险验证 + 处置闭环”的产品。

本方案明确结论：

- 仅靠继续追加页面和筛选，不足以达到目标。
- 必须重构扫描数据模型。
- 必须重构主机漏洞扫描主路径。
- 必须重构网站扫描的认证、发现与验证链路。
- 漏洞知识库不再作为平台内在线同步功能，而是改为“平台外同步，平台内导入”。

## 参考思路

参考绿盟 RSAS 公开资料，提炼出以下产品原则：

1. 先资产、后风险
   - 先发现资产、服务、Web 攻击面，再做漏洞与配置检查。

2. 风险以验证结果为主
   - 真正进入报表、风险评分和处置流程的，应是已验证风险，而不是所有推断线索。

3. 主机与 Web 都要支持认证后检查
   - 主机侧不仅有端口与漏洞，还应包括登录后核查、弱口令、配置检查。
   - Web 侧不仅有匿名扫描，还应覆盖登录态、会话维持、复杂认证和登录后页面。

4. 误报修正和验证流程必须直观
   - 用户不应理解过多内部引擎概念。
   - 前台默认只看到“正式结果 / 待验证 / 资产信息”和直观的验证状态。

5. 扫描结果必须可回溯
   - 每次运行应有独立快照、独立证据、独立统计。

## 当前系统现状

当前系统已经具备可运行骨架：

- 主机侧：
  - `Nmap` 发现
  - 版本/CPE 匹配知识库
  - `Nuclei` 服务模板验证
- Web 侧：
  - URL/页面发现
  - `web-template`
  - `web-rule`
- 结果层：
  - `正式结果`
  - `待验证`
  - `资产信息`
- 数据层：
  - 已补 `task/run` 的最小基础字段
  - 已补结果来源、匹配方式、知识库关联、验证状态等语义字段

但距离成熟扫描平台仍有结构性差距：

1. 主机扫描仍以“版本推断 + 模板命中”为主，缺少登录后检查和配置核查。
2. Web 扫描的复杂认证和登录后业务链路支持偏弱。
3. `engine.go` 仍然承担过多执行职责，主机和 Web 主路径耦合过深。
4. 结果与证据模型仍不够细，尚未完全升级到“运行快照 -> 目标单元 -> 证据 -> finding occurrence”的形态。
5. 验证、误报修正、例外范围等流程还不够产品化。

## 必须重构的边界

以下部分建议明确按“重构”处理，而不是继续小修小补：

### 1. 扫描数据模型

必须从“任务直写结果”升级为：

- `scan_tasks`
  - 长期任务定义
- `scan_runs`
  - 每次运行快照
- `scan_targets`
  - 本次运行展开出的 IP、服务、URL、接口、页面
- `scan_evidences`
  - 原始证据
- `scan_findings`
  - 稳定风险对象
- `scan_finding_occurrences`
  - 某次运行中的实际命中记录

现有 `task_id` 直接挂到漏洞记录上的模型，只能继续兼容，不应作为长期核心模型。

### 2. 主机漏洞扫描主路径

必须从“发现 -> 版本匹配 -> 模板验证”的单流水线，升级为：

- 主机发现
- 服务识别
- 资产归一
- 版本线索生成
- 协议/模板验证
- 认证后检查
- 配置核查
- 风险归并

### 3. 网站漏洞扫描主路径

必须从“入口发现 + 模板/规则”升级为：

- 匿名攻击面发现
- 登录态获取与会话维持
- 页面/API/表单/鉴权入口图构建
- Web 模板扫描
- 规则扫描
- 登录后页面扫描
- 认证/授权类专项验证
- 风险归并与证据沉淀

## 目标架构

### 一、统一扫描架构

所有扫描统一走以下层次：

1. 任务定义层
   - 只描述“想扫什么”

2. 运行快照层
   - 每次执行独立快照

3. 目标展开层
   - 主机、服务、URL、页面、接口、表单都变成标准目标对象

4. 检测引擎层
   - 资产识别
   - 版本线索
   - 模板验证
   - 登录后检查
   - 配置核查

5. finding 归一层
   - 统一输出正式结果、待验证、资产信息

6. 验证/误报修正层
   - 统一处理验证状态、误报、忽略、例外范围

### 二、前台默认只保留三类结果

- `正式结果`
  - 真正进入风险评分、导出报告、处置流程
- `待验证`
  - 基于版本、资产和规则推断得到的风险线索
- `资产信息`
  - 端口、服务、版本、枚举、技术栈、暴露面事实

## 主机扫描重构目标

### 目标能力

主机侧目标不应只停在漏洞扫描，而应包含：

- 端口发现
- 服务识别
- 资产归类
- 版本线索
- 协议/模板验证
- 弱口令检测
- 登录后检查
- 安全配置核查
- 数据库/中间件专项检查

### 推荐主机扫描执行流

```text
目标输入
  -> 主机发现
  -> 服务识别
  -> 服务实例归一
  -> 版本线索生成
  -> 协议模板验证
  -> 认证后检查
  -> 配置核查
  -> 证据落库
  -> finding 归并
  -> 风险统计
```

### 主机侧结果分类

- `资产信息`
  - 端口
  - 服务
  - 产品
  - 版本
  - banner
- `待验证`
  - 版本/CPE 命中知识库
  - 弱证据规则命中
  - 未完成验证的风险线索
- `正式结果`
  - 协议验证命中
  - 认证后检查命中
  - 配置核查失败
  - 手工确认转正

### 主机侧优先改造点

1. 拆分 `engine.go` 主机扫描主流程
   - 抽成 `discover -> correlate -> verify -> audit -> persist`

2. 引入登录后检查插件层
   - 先从 SSH / WinRM / SMB / 数据库凭据检查开始

3. 引入配置核查模型
   - 将“弱配置”从普通漏洞里分离成专项能力

4. 将版本线索彻底内部化
   - 允许作为待验证结果存在
   - 不再成为前台必须理解的引擎概念

## 网站扫描重构目标

### 目标能力

Web 侧目标应包含：

- 匿名扫描
- 登录后扫描
- SPA/接口型应用发现
- 表单与接口攻击面发现
- 多认证模式支持
- 动态签名/多头会话支持
- 页面、接口、表单三类对象统一建模

### 推荐网站扫描执行流

```text
目标 URL
  -> 基础探测
  -> 页面/API/表单发现
  -> 登录态获取
  -> 会话维持
  -> 登录后继续发现
  -> Web 模板扫描
  -> 规则扫描
  -> 证据落库
  -> finding 归并
```

### Web 侧重点重构项

1. 认证引擎重构
   - 从平铺字段升级为统一认证计划
   - 支持多头、Cookie、Token、Basic、登录表单、登录接口、动态签名

2. 浏览器发现与 HTTP 发现统一
   - 页面发现不是浏览器路径和 HTTP 路径割裂
   - 统一沉淀成攻击面图

3. 登录后扫描成为一等能力
   - 不是“有 Header 就算支持认证”
   - 需要能真正覆盖登录后的页面、接口和表单

4. Web finding 模型升级
   - 页面、接口、表单、认证入口要能单独追踪
   - 证据与上下文可回溯

## 结果与验证模型

### 统一状态

建议后续收敛为：

- `未验证`
- `复测中`
- `验证成功`
- `验证失败`
- `误报`
- `已忽略`

### 验证与误报修正

参考成熟产品思路，后续应支持：

- 对单条结果标记验证状态
- 对某主机范围标记误报
- 对某任务范围标记误报
- 对某规则/模板范围配置例外

## 漏洞知识库策略

### 原则

- 平台内不再直接在线同步 NVD/CNVD/CNNVD
- 平台外完成同步、清洗、去重、格式转换
- 平台内只负责导入、浏览、检索和关联

### 推荐流程

```text
外部同步器
  -> 拉取 NVD/CNVD/CNNVD
  -> 清洗和标准化
  -> 生成 CSV 数据包
  -> 平台内导入
  -> 更新本地知识库
```

### 原因

- 降低平台运行时外部依赖
- 避免长时间后台同步任务和失败重试复杂度
- 避免“同步中”状态长期占据产品面
- 让知识库变成稳定基线数据，而不是运行时行为

## 分阶段路线图

### Phase 0：冻结现状语义

目标：

- 前台统一结果语义
- 下线知识库在线同步
- 固化“平台外同步，平台内导入”

完成标准：

- 不再向用户暴露内部扫描概念
- 知识库页以导入和浏览为主

### Phase 1：数据模型重构

目标：

- 完成 `task/run/target/evidence/finding_occurrence` 基础表设计
- 保留现有结果表作为兼容层

关键产物：

- 新表迁移
- 运行快照模型
- 证据表模型
- finding occurrence 模型

### Phase 2：主机扫描重构

目标：

- 拆主机扫描执行器
- 接入认证后检查和配置核查
- 让主机扫描从“版本推断平台”升级为“主机风险评估平台”

关键产物：

- `host_discovery`
- `host_correlation`
- `host_verification`
- `host_authenticated_audit`
- `host_posture_profiles`

### Phase 3：网站扫描重构

目标：

- 接入统一认证计划
- 重构攻击面发现
- 支持登录后扫描和复杂认证

关键产物：

- `web_auth_engine`
- `web_surface_graph`
- `web_scan_runner`
- `request_signer`

### Phase 4：验证与误报修正产品化

目标：

- 统一验证状态
- 增加误报修正和例外范围
- 报告和详情页按作用域展示

### Phase 5：报告、审计和运营能力收口

目标：

- 报告体系升级
- 历史对比
- 趋势分析
- 资产与风险联动

## 推荐实施顺序

按性价比和风险排序，建议这样推进：

1. 先完成数据模型重构
2. 再重构主机扫描
3. 再重构 Web 认证与发现
4. 最后做验证、误报修正和报告升级

原因：

- 不先改模型，后面所有扫描能力都会继续堆到旧结构里。
- 主机侧改造更容易验证收益，也更容易先形成稳定阶段成果。
- Web 侧复杂认证和发现链路最难，适合放在模型稳定之后处理。

## 立即执行建议

下一步建议直接启动以下 3 项：

1. 出一版新的扫描数据模型迁移设计
   - 明确 `run / target / evidence / occurrence`

2. 将主机扫描从 `engine.go` 中彻底拆出独立执行器
   - 不再继续在超长函数里堆逻辑

3. 出一版 Web 统一认证计划设计
   - 替代现有平铺认证字段

## 结论

如果目标是参考绿盟思路，把当前安全中心升级成更成熟的漏洞扫描平台，那么：

- 主机漏洞扫描需要重构
- 网站漏洞扫描需要重构
- 数据模型必须先重构

继续在现有模型上补功能，只能得到“更复杂的当前版本”，得不到真正稳的下一代扫描平台。
