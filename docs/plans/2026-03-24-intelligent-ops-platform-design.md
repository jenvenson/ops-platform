# 智能化运维平台总体设计方案

## 1. 文档目标

本文档用于定义 OPS Platform 向“智能化运维平台”演进时的总体设计原则、数据闭环、决策闭环和分期建设路径。

本文档解决 4 个问题：

- 平台未来要智能化到什么程度
- 智能能力应建立在什么数据底座上
- 每类智能能力如何形成决策闭环
- 一期、二期、三期分别做什么

设计基线：

- 先设计数据和决策闭环，再设计模型接入
- 先做辅助智能，再做执行智能
- 模型不直接连生产系统，统一走受控工具层
- 高风险动作必须鉴权、确认、审计
- 智能能力必须可解释、可回溯、可降级

## 2. 平台目标

### 2.1 总体目标

将 OPS Platform 从“功能集成平台”升级为“具备认知、分析、建议和受控执行能力的智能化运维平台”。

平台最终应具备以下能力：

- 能理解用户问题和平台事件
- 能从资产、配置、部署、告警、安全等数据中提取上下文
- 能生成有依据的建议，而不是只给文本回答
- 能将建议转为受控动作，并形成执行回写
- 能对所有智能分析和执行过程进行审计

### 2.2 近期目标

在当前阶段，平台智能化建设优先解决以下问题：

- 用户找不到页面入口、操作路径不清晰
- 问答和平台实时状态之间仍然割裂
- 发布、告警、漏洞、配置变更之间缺少关联分析
- 平台缺少统一的“建议 -> 确认 -> 执行 -> 回写”闭环

## 3. 建设原则

### 3.1 数据优先

平台智能化首先是数据工程问题，其次才是模型工程问题。

必须先保证：

- 关键业务对象定义统一
- 关键事件能挂接到对象
- 对象之间关系可查询
- 事件能沉淀成时间线
- 所有分析结果都能回写

### 3.2 辅助优先

平台先做“辅助决策”，不直接追求“全自动执行”。

优先级顺序：

1. 可回答
2. 可检索
3. 可分析
4. 可建议
5. 可受控执行

### 3.3 治理内建

权限、审计、风险分级、降级策略必须内建在架构中，不能等功能做完后补。

## 4. 智能化能力分层

建议将平台智能能力划分为 4 层。

### 4.1 知识层

负责沉淀静态和半静态知识：

- 产品文档
- 用户手册
- SOP
- 发布规范
- 安全规范
- 常见问题与修复经验

### 4.2 数据层

负责沉淀实时业务数据和关系数据：

- CMDB 资产数据
- 发布与归档记录
- Jenkins 构建结果
- Consul 配置变更
- 告警事件与规则
- 安全漏洞与工单
- 平台用户与权限

### 4.3 决策层

负责对知识和实时数据做组合分析：

- 检索
- 归纳
- 关联分析
- 风险判断
- 候选动作生成
- 解释与依据输出

### 4.4 执行层

负责承接决策结果并执行受控动作：

- 页面导航
- 只读查询
- 半自动执行
- 审批确认
- 结果回写

## 5. 数据闭环设计

### 5.1 核心对象模型

平台至少统一这 4 类核心对象。

#### 资产对象

- 项目
- 环境
- 主机
- 应用
- Jenkins 流水线
- Consul 配置
- 安全资产

#### 事件对象

- 部署事件
- 归档事件
- 聚合打包事件
- 构建事件
- 告警事件
- 配置变更事件
- 漏洞事件
- 工单事件
- 会话事件

#### 关系对象

- 应用属于哪个项目
- 应用部署到哪些环境
- 告警关联哪些主机或应用
- 漏洞影响哪些资产
- 发布影响哪些配置或服务

#### 时间线对象

每个核心对象都应支持时间线视图，例如：

- 某应用最近 7 天的部署、告警、配置变更、漏洞、工单
- 某主机最近 24 小时的告警与发布影响
- 某项目在一次变更窗口内发生过哪些关键事件

### 5.2 数据闭环要求

每条事件都至少需要包含：

- `event_type`
- `object_type`
- `object_id`
- `related_object_ids`
- `operator_id`
- `status`
- `started_at`
- `finished_at`
- `summary`
- `metadata`
- `source`

每条智能分析结果也应视为事件，并回写到事件流中。

### 5.3 统一事件流

建议补一层平台事件流抽象，统一承接不同模块输出。

```text
CMDB / Deploy / Jenkins / Consul / Alarm / Security / Assistant
  -> Event Normalizer
    -> Unified Event Store
      -> Object Timeline
      -> Search / Analytics / Assistant
```

作用：

- 为智能分析提供统一上下文
- 为审计提供统一回放能力
- 为后续推荐、预测、自动化提供输入

## 6. 决策闭环设计

### 6.1 决策闭环模板

每一类智能能力都必须遵循统一闭环：

```text
触发输入
  -> 数据获取
    -> 检索/规则/模型分析
      -> 决策输出
        -> 用户确认/审批
          -> 受控执行
            -> 结果回写
```

### 6.2 触发输入

触发源分为 3 类：

- 用户主动提问
- 页面上下文触发
- 系统事件触发

示例：

- 用户问“如何查看归档历史”
- 用户在部署失败页面点击“智能分析”
- 系统检测到高危漏洞新增，自动生成建议

### 6.3 数据获取

由统一工具层完成，禁止模型直接访问底层系统。

典型工具：

- 文档检索工具
- 页面导航工具
- 发布历史查询工具
- 告警事件查询工具
- 漏洞查询工具
- 配置变更查询工具
- 资产关系查询工具

### 6.4 分析与决策

决策层应同时支持：

- 规则判断
- 检索增强
- 大模型总结
- 规则与模型融合

输出必须结构化，至少包括：

- `decision_type`
- `summary`
- `evidence`
- `risk_level`
- `candidate_actions`
- `need_confirmation`
- `confidence`

### 6.5 受控执行

执行动作必须分级：

#### L1 建议

- 只返回建议，不执行

#### L2 半自动

- 生成动作
- 用户确认后执行

#### L3 受控自动

- 仅对低风险动作开放
- 必须满足规则条件
- 必须保留审计记录

当前阶段建议平台默认只做到 `L2`。

### 6.6 结果回写

执行或建议完成后，统一回写到：

- assistant 会话记录
- 业务事件流
- 审计日志
- 对象时间线

回写后，下一轮分析才能真正具备上下文连续性。

## 7. 架构设计

### 7.1 总体架构

```text
前端交互层
  -> Assistant / 页面建议入口 / 分析卡片

决策编排层
  -> 意图识别
  -> 上下文组装
  -> 工具编排
  -> 规则引擎
  -> 模型调用
  -> 输出格式化

工具层
  -> 文档检索工具
  -> 页面导航工具
  -> CMDB 查询工具
  -> 发布查询工具
  -> 告警查询工具
  -> 安全查询工具
  -> 执行动作工具

数据底座
  -> 统一对象模型
  -> 统一事件流
  -> 时间线索引
  -> 文档知识库
  -> 审计日志
```

### 7.2 模型接入位置

模型属于决策编排层的一部分，不属于平台核心业务层。

接入原则：

- 模型不直接发起执行
- 模型不直接拼接 SQL 或 Shell
- 模型输出必须经过规则校验
- 模型失败时要能回退到规则或静态回答

## 8. 一期到三期路线图

### 8.1 一期：助手化

目标：让平台“能回答、能导航、能查询”。

能力范围：

- 知识问答
- 页面导航
- 只读查询
- 会话管理
- 结构化引用
- 基础验收体系

验收标准：

- 高频问法能返回正确路径和步骤
- 回答有真实引用来源
- 只读查询可用
- 会话能力完整

### 8.2 二期：分析化

目标：让平台“能关联、能解释、能建议”。

能力范围：

- 部署失败分析
- 告警归因分析
- 漏洞优先级判断
- 配置变更影响分析
- 发布前风险提示

验收标准：

- 能将多个模块数据拼接为统一结论
- 能给出明确依据
- 能输出候选动作而非纯文本说明

### 8.3 三期：执行化

目标：让平台“能受控执行”。

能力范围：

- 半自动发布
- 半自动回滚
- 半自动配置下发
- 工单联动
- 审批闭环

验收标准：

- 所有高风险动作都有确认与审计
- 执行结果自动回写事件流
- 模型不可用时仍可退回规则执行路径

## 9. 关键治理要求

### 9.1 权限

- assistant 工具调用必须继承当前用户权限
- 不允许绕过现有 RBAC
- 不同角色只能看到允许的数据范围

### 9.2 审计

必须记录：

- 用户原始输入
- 使用的引用和工具
- 生成的建议
- 最终执行动作
- 执行结果

### 9.3 风险分级

建议至少分为：

- `low`
- `medium`
- `high`
- `critical`

其中：

- `high` 以上动作默认禁止自动执行
- 删除、回滚、批量配置变更默认不进入自动执行

### 9.4 降级

在以下情况必须支持降级：

- 模型不可用
- 检索失败
- 工具超时
- 数据源不可达
- 权限不足

降级策略：

- 返回规则化回答
- 返回静态知识
- 返回人工确认建议

## 10. 当前代码落点建议

在现有代码基础上，建议按以下方向演进：

- `backend/internal/assistant/`
  继续承接智能交互主链路
- `backend/internal/*`
  各业务模块继续提供受控查询与执行能力
- 新增统一事件抽象层
- 新增对象关系与时间线查询层
- 保持前端 `AIChatbot` 为智能交互主入口，同时逐步增加页面内“智能分析”入口

## 11. 下一步设计产物

建议按顺序继续输出以下文档：

1. `统一对象模型设计`
2. `统一事件流与时间线设计`
3. `assistant 决策编排设计`
4. `二期智能分析场景清单`
5. `执行动作风险分级与审批设计`

## 12. 结论

OPS Platform 的智能化演进，不应从“接入大模型”开始，而应从“建立统一数据闭环和决策闭环”开始。

本阶段最重要的不是让模型更聪明，而是确保：

- 数据可追踪
- 决策可解释
- 动作可控制
- 结果可回写

当这套底座稳住以后，再增加 RAG、规则引擎、模型编排和受控执行，平台才会真正走向“智能化”，而不是停留在“聊天化”。
