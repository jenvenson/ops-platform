# AGENTS.md

## Purpose
- 默认先理解现有代码、配置和部署方式，再做修改。
- 优先最小必要改动，避免顺手重构。
- 修改后必须验证；无法验证时明确说明缺口。
- 不清楚的地方先查证，不要基于猜测修改。

## Scope
- 处理当前 worktree 时，优先以当前工作目录对应的项目为准。
- 若仓库存在多层 AGENTS.md，遵循“越接近当前目录，优先级越高”的规则。
- 如遇目录归属不清，先确认当前目录、Git 根目录、实际服务启动目录，再继续修改。

## Repo Facts
- 主工程目录：`backend/`、`frontend/`、`deploy/`、`docs/`
- `_archive/` 不参与主运行链路，默认不作为分析入口。
- 后端入口：`backend/cmd/server/main.go`
- 前端 API 客户端：`frontend/src/api/client.ts`
- 小助手主链路：`/api/assistant/*`
- Ollama 当前永久禁用；后续如重新启用，再补充说明
- `docker-compose restart` 不会重建容器

## Workflow
1. 先定位入口、调用链、配置来源和实际生效点。
2. 优先阅读主干文件，不从 `_archive/` 开始。
3. 先确认问题发生层：前端、后端、接口、模型配置、部署配置。
4. 做最小必要修改，不顺手改无关代码。
5. 修改后执行直接相关验证。
6. 回复时先给结论，再给证据、影响面、风险和未验证项。

## Change Rules
- 不要为了“看起来更干净”而做额外重构。
- 不要主动改动无关文件、命名风格或目录结构。
- 需要跨模块修改时，先说明为什么不能局部修复。
- 涉及配置项时，先确认“定义位置、读取位置、生效位置”三处。
- 涉及 assistant 改动时，同时检查：
  - 请求入口
  - 模型配置来源
  - 实际调用模型名
  - 落库模型名
  - 返回给前端的数据结构

## Safety Rules
- 不要执行 `git reset --hard`、`git checkout --`。
- 不要覆盖用户已有改动，除非明确要求。
- 不要直接改线上数据库结构。
- 不要直接删除库表、历史数据、生产配置。
- 涉及线上操作时，先确认目标机器、目录、环境和影响面。
- 涉及 Docker / Compose / K8S 操作时，先说明是否会重启、重建或中断服务。

## Logging Rules
- `./logs/` 是默认任务日志目录，不要把过程记录散落到仓库根目录。
- 每次开始处理一个新任务时，先执行 `mkdir -p logs`。
- 除纯只读且瞬时完成的短命令外，默认记录命令日志到 `logs/<task>.log`。
- 运行耗时较长、排障、测试、构建、联调、扫描、批处理命令时，必须记录日志到 `logs/<task>.log`。
- 需要同时记录 stdout 和 stderr。
- 优先使用：
  - `mkdir -p logs`
  - `<command> 2>&1 | tee -a logs/<task>.log`
- 需要尽量实时刷盘时，优先使用：
  - `stdbuf -oL -eL <command> 2>&1 | tee -a logs/<task>.log`
- 仅在以下场景额外维护简要过程记录文件 `logs/<task>.md`：
  - 跨模块或多阶段任务
  - 排障与事故定位
  - 需要后续交接或复盘的任务
  - 用户明确要求保留过程记录
- 若需要写 `logs/<task>.md`，建议包含：
  - 背景 / 目标
  - 关键判断
  - 执行过的关键命令
  - 修改的文件
  - 验证结果
  - 未解决问题 / 风险
- 不要求保留逐次思考过程；除上述场景外，不强制新增过程记录文件。
- 若已有同名任务日志，优先追加，不要覆盖；除非明确需要，不删除历史日志。
- 定期归档旧日志：
  - 默认只保留当前活跃任务相关日志
  - 跨版本、跨阶段或超过一个月的旧日志，优先移动到 `logs/archive/<yyyy-mm>/`
  - 若日志只用于临时排障且问题已收口，可在确认无追溯需求后直接删除，不强制长期保留
- 回复用户前，确认本次任务至少满足其一：
  - 已为关键命令留下 `logs/<task>.log`
  - 若属于重要任务 / 排障 / 需交接任务，已补 `logs/<task>.md`

## Validation
- 后端改动优先检查：
  - 路由是否注册
  - 配置是否读取
  - 日志是否出现
  - 接口返回结构是否符合预期
- 前端改动优先检查：
  - 页面入口
  - API 调用参数
  - API 返回结构适配
  - 关键交互是否报错
- 涉及 assistant 改动时，至少核对：
  - 模型配置值
  - 实际运行日志中的模型名
  - 数据库落库模型名
  - 前端展示模型名（如果有）
- 能跑测试就直接跑最相关测试；不能全量测试时，至少做最小闭环验证。
- 自动判断是否需要运行 smoke：
  - 默认运行 smoke 的场景：
    - 修改前端路由、菜单、入口、跳转逻辑
    - 修改用户可直接触达的主流程，如登录、发布、扫描、查询、详情、右上角入口
    - 修改运维小助手的页面导航、动作按钮、导航词映射
    - 修改页面标题、关键按钮、关键筛选项等高可见内容，且影响多个页面或主流程
    - 跨多个前端页面或多个模块的联动修改
  - 默认不运行 smoke 的场景：
    - 纯后端逻辑改动，且未直接影响前端主流程
    - 纯文档改动
    - 很小的局部重构、注释调整、非行为变更
    - 更适合用单测、接口测试、SQL 校验验证的改动
  - 若不跑 smoke，应选择更贴近改动面的验证方式，如单测、接口测试、数据校验或人工 spot check。
  - 若改动明显影响用户主流程，但未运行 smoke，回复中必须说明原因。
- 无法验证时，明确写出：
  - 为什么不能验证
  - 缺少什么条件
  - 哪些风险尚未排除

## Preferred Commands
- 修改前优先使用只读命令理解项目，如：
  - `pwd`
  - `git status`
  - `find`
  - `grep` / `rg`
  - `ls`
- 修改后优先执行与改动直接相关的最小验证，不默认跑高成本全量流程。
- 除非明确需要，不做全仓大范围格式化。

## Response Format
- 结论：一句话说明修了什么 / 发现了什么。
- 证据：列出关键文件、关键配置、关键日志或验证结果。
- 影响面：说明会影响哪些模块。
- 风险：说明未验证项、潜在副作用、上线注意点。
